# Session 2 challenge box

## Programming Challenges

### 🟢 Easy
- You have a list of size `n`. You loop once to replace all `""` with `"N/A"`. What is the time complexity and why?  
- You create a copy using `new_data = my_data.copy()`. What are the time and space complexities? Explain.  
- You search for a value in an unsorted list using a loop. What is the time complexity and why?  

### 🟡 Normal
- Compare cleaning data **in place** vs **copying first**. What are the time and space complexities for each?  
- You sort a list and then perform binary search. What is the total time complexity? When is this worth it?  
- You count missing values in a list of size `n` using one pass. What is the time complexity? What if you repeat this operation multiple times?  

### 🔴 Difficult
- You read a dataset with 10 million rows into memory and clean it. What are the time and space complexities? What risks arise?  
- You process a large file line-by-line instead of loading it fully. How does this affect space complexity and performance?  
- You clean data in place but the program crashes midway. What are the consequences? How would copying first change this?  

### 🔥 Hell
- You need to support **fast repeated searches** on a dataset of size `n`. You can either:
  1. Search the unsorted data each time, or  
  2. Sort the data once and then use binary search.  

  What is the time complexity of each approach, and after how many searches does sorting become the better option?
  
## Indicative Answers

### 🟢 Easy
- Single loop replace → **O(n)** time, because each element is visited once.  
- Copy list → **O(n)** time and **O(n)** space, because all elements are duplicated.  
- Linear search → **O(n)** time, worst case checks every element.  

### 🟡 Normal
- In-place: **O(n)** time, **O(1)** extra space.  
  Copy-first: **O(n)** time, **O(n)** space. Trade-off = safety vs memory.  
- Sort + binary search → **O(n log n + log n) ≈ O(n log n)**. Worth it if many searches follow.  
- Counting once → **O(n)**. Repeating `k` times → **O(kn)** (inefficient, can store result instead).  

### 🔴 Difficult
- Load + clean → **O(n)** time, **O(n)** space. Risk: memory exhaustion, data corruption if modified in place.  
- Line-by-line → **O(n)** time, **O(1)** space. Slower per operation but scalable for large data.  
- In-place crash → partial corruption, hard to recover. Copy-first → original preserved, safer recovery.  

### 🔥 Hell
- Searching unsorted each time → **O(n)** per search → total **O(k·n)** for `k` searches.  

- Sort once + binary search:
  - Sorting → **O(n log n)**  
  - Each search → **O(log n)**  
  - Total → **O(n log n + k·log n)**  

- Sorting becomes beneficial when:
  ```text
  k·n > n log n + k·log n
  ```
  That simplifies to `k > log n`
  
  So, if the number of searches is greater than log n, sorting first is the better choice.
